# 2.17.0

`render_audio` の長時間実行に対する応答方式（ジョブ方式）を整備する minor release。

## 追加機能

- `render_audio` に `background` を追加する。`background:true` で `Resources.renderPreFxAudio` を待たずに開始し、`{ status:"started", jobId }` を即時返す。完了時にジョブ状態へ反映する。
- `get_render_job`: `jobId` でレンダリングジョブの状態（`running` / `done` / `error`）を照会する。`done` で `filePath`、`error` で `error` を含む。存在しない jobId は `not_found`。

## 判断根拠（記録）

- MCP トランスポートはステートレスな Streamable HTTP（リクエストごとに新規セッション）で progress notification を送れないため、長尺レンダリングは MCP クライアントのツール呼び出しタイムアウトに達し得る。所要時間は Live 実機依存で本リポジトリでは実測できない。
- そこで実測に依存せず結果を確定させるため、ジョブ方式を採用する。短尺は既定の同期モード（`filePath` を返す）で完結し、長尺は `background:true` で `jobId` を返して `get_render_job` で照会する。これにより結果は「完了（filePath）・明示エラー・照会可能なジョブ」のいずれかに必ず確定する。
- ジョブ状態は module singleton の `Map` に最新 `MAX_RENDER_JOBS`(50) 件を保持する。ホスト再起動でジョブは失われる（`get_render_job` は `not_found`）。

## 動作確認

- `pnpm test`: `background:true` で started→running→done（filePath 反映）と、未知 jobId の `not_found` をフェイク context で検証。
- `pnpm typecheck`: 全パッケージ成功。既定（同期）モードの挙動は不変。

## 制約

- 実際のレンダリング所要時間・主要 MCP クライアントのタイムアウトの実測は Live 実機が必要であり、本リリースには含まない。ジョブ方式は実測に依存せずタイムアウト不定を解消する。
- 関連: #21（render_audio 導入）、#24（近似レンダリング経路を実装する場合は同じジョブ方式を共有できる）。
