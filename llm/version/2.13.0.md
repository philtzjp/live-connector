# 2.13.0

複数の書き込みを単一トランザクションで実行する `batch` ツールを追加する minor release。

## 追加機能

- `batch`: `set_song` / `set_track` / `set_clip` / `set_scene` / `set_cue_point` / `set_device_parameter` / `write_notes` のステップ列を 1 回の呼び出しで実行し、Live の undo 履歴上 1 ステップにまとめる。`preview` で解決済みプラン、`confirm` で大量ターゲット許可。
- 全ステップを適用前に解決・検証する。解決・検証段階で失敗した場合は何も適用せず、`failedStep` と理由を返す。（2.17.8 追記: 当初記載の「all-or-nothing」「入力検証は単体ツールと同一」は適用段階の失敗と per-property 検証で成立していなかった。#87 / v2.17.8 で、入力検証を単体ツールと同一の zod スキーマに揃え、適用段階の失敗は appliedSteps / failedSteps / unappliedSteps で識別する仕様へ修正。）
- `batch` の実行は書き込み履歴（#57）にも記録される。

## 変更内容

- `withinTransaction` のコールバックは同期必須で内部 await 不可のため、batch は「対象を先に非同期解決 → 全ミューテーションを 1 つの同期 `withinTransaction` コールバックで初期化（`Promise.all`）」する。ネストは最外へ統合され 1 undo ステップになる。
- 制約: 同期コールバック内で await できないため、後続ステップは同一 batch 内で先行ステップが作成したオブジェクトを参照できない。構造操作（create / delete / duplicate）は本ツールの対象外とする。
- `notes.ts` の `noteSchema` / `toNoteDescription` を再利用のため export。`llm/models.yaml` / `llm/ARCHITECTURE.md` を更新。

## 動作確認

- `pnpm test`: batch の preview（未適用）・単一 undo での適用・解決失敗時の all-or-nothing をフェイク SDK Song で検証。
- `pnpm typecheck`: 全パッケージ成功。

## 制約

- 対象ステップは「解決後に同期適用できる」書き込み（set_* / write_notes）に限る。create / delete / duplicate / load_sample / render_audio は非対応（SDK の同期トランザクションコールバック制約による）。
- 関連: #50（巻き戻し）、#48（読み取りの集約に対する書き込み側の集約）。
